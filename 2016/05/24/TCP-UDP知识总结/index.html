TCP/UDP知识总结

发表于 2016-05-24   |   分类于 技术   |  

TCP和UDP是计算机网络中非常重要的两个协议，他们都位于传输层。TCP用于提供可靠的通信传输，而UDP则常被用于让广播和细节控制交给上层的应用去处理。在网络层的IP数据报的首部有一个协议字段，用来标识网络层的上一层所采用的是哪一种传输协议。

传输层的任务

传输层的上层是应用层，传输层的任务就是把数据交给上面的应用，通常一台计算机运行着多个应用程序，因此有必要将它们区分开来。为了实现这种功能，使用端口号这样一种识别码，不同的应用程序有不同的端口号。

通信处理

我们通常将通信双方中发起请求的一方称为客户端，处理发送方的请求的一方叫做服务端。服务端的程序有必要提前启动，否则，即使有客户端的请求发送过来，也无法作出相应的处理。这些服务端程序在UNIX系统中叫做守护进程，例如HTTP的服务端程序是httpd（HTTP守护进程(damon)），而ssh的服务端程序是sshd（SSH守护进程）。在UNIX中并不需要将这些守护进程逐个启动，而是启动一个可以代表他们接受请求的inetd（互联网守护进程）服务程序即可。这是一种超级守护进程，由它可是新创建进程并将这个新进程转换成特定的守护进程。确定一个请求究竟发给的是哪个服务端守护进程，可以通过检查数据报中的目标端口号来识别。目标端口为22，则转给sshd，为80，则转给httpd。

TCP和UDP的区别

TCP

TCP是面向连接的、可靠的数据流协议。由于它是可靠的，所以需要有诸如“顺序控制”、“重发控制”、“拥塞控制”等机制。

UDP

UDP是不具有可靠性的数据报协议。细微的处理它会交给上层的应用去处理。它不保证发出去的数据报一定会到达接收端。

但是，这并不是说，TCP一定优于UDP。因为有的应用场景对可靠性要求本来就不高，所以他们更适合用UDP传输协议。例如，日常生活中打电话，这种应用场景对高速传输和实时性有较高要求，声音会被转换成数字信号传输过去，如果采用TCP，丢失的数据报必须重发，那么用户就会感到卡顿，影响到用户交谈，而其实丢失一些少数的数据报其实并无大碍，不影响人们的正常交流。所以应该选用UDP而不是TCP。

UDP

UDP不提供复杂的控制机制，利用IP提供面向无连接的通信服务，并且它是在将应用程序发来的数据在收到的那一刻，立即按照原样发送到网络上的一种机制。

即使在出现网络拥堵的情况下，UDP也无法进行流量控制等避免拥塞的行为。此外，传输过程中即使出现丢包，UDP也不负责重发，甚至当到达接收方时数据报的顺序乱掉也没有纠正的功能，如果需要实现这些细节控制，只有在上层的应用程序中去处理。

由于UDP是面向无连接的，它可以随时发送数据。再加上UDP简单高效，因此它常用于以下几个方面：

  • 包总量较少的通信（DNS服务等）
  • 视频、音频等多媒体通信
  • 限定于LAN等特定网络中的应用通信
  • 广播通信

TCP

TCP通过检验和、序列号、确认应答、重发控制、连接管理以及窗口控制等机制实现可靠传输。

在TCP中，当发送端的数据到达接收主机时，接收端主机会返回一个已收到消息的通知，这个消息叫做确认应答（ACK）。

TCP在发出数据报后会“期待”收到一个回应，如果收到回应，则说明数据已经成功送到接收端，若没有收到回应，则有可能发生数据丢失。当发送方认为发生数据报丢失时，进行重发。

未收到确认应答并不意味着真的发生数据丢失，有可能是确认应答在返回的过程中丢失了。

此外，有可能重发之后，确认应答才到达发送方。这种情况下，接收方就会收到相同的数据。显然应该讲重复的数据报丢掉，因为TCP是可靠的传输，它应该向上层传递与发送方完全一样的数据包括顺序、数量。为了解决这种问题，引入了序列号的概念，序列号是把发送的每一个字节都标上号码的一个编号。接收端查询接收到的数据TCP首部中的序列号和数据的长度，将自己下一步想要接收的数据的序列号作为确认应答返回去。通过序列号和确认应答，TCP就可以实现可靠传输了。

重发时间

重发时间其实并不好确定，如果重发时间过短，那么那些并没有发生数据报丢失的情况也会重发，如果重发时间过长网络吞吐量就会下降。而且确认应答的返回时间会受网络拥堵情况的影响。为此，在每次发包之前它都会计算往返时间和偏差（波动值）。这里有好几个算法，这里就不一一展开了，想要深入了解的参看这个链接

在BSD的UNIX和Windows中，超时都是以0.5秒为单位进行控制，重发时间都是0.5秒的整数倍，由于最开始的数据报还不知道往返时间，所以重发超时一般设置为6秒左右。
数据被重发之后若还是收不到确认应答，则进行再次发送，此时等待确认的时间会以2倍、4倍呈指数增长。但是不会无限的重发，当重发达到一定次数之后，则认为网络或接收方的发生了异常，这时强制关闭连接。

TCP报文格式


简要说明部分字段代表的含义：
Source Port和Destination Port:分别占用16位，表示源端口号和目的端口号；用于区别主机中的不同进程，而IP地址是用来区分不同的主机的，源端口号和目的端口号配合上IP首部中的源IP地址和目的IP地址就能唯一的确定一个TCP连接；

  • Sequence Number:用来标识从TCP发端向TCP收端发送的数据字节流，它表示在这个报文段中的的第一个数据字节在数据流中的序号；主要用来解决网络报乱序的问题；
  • Acknowledgment Number:32位确认序列号包含发送确认的一端所期望收到的下一个序号，因此，确认序号应当是上次已成功收到数据字节序号加1。不过，只有当标志位中的ACK标志（下面介绍）为1时该确认序列号的字段才有效。主要用来解决不丢包的问题；
  • TCP Flags:TCP首部中有6个标志比特，它们中的多个可同时被设置为1，主要是用于操控TCP的状态机的，依次为URG，ACK，PSH，RST，SYN，FIN。标志位的意思如下：
    • URG：此标志表示TCP包的紧急指针域有效，用来保证TCP连接不被中断，并且督促中间层设备要尽快处理这些数据；
    • ACK：此标志表示应答域有效，就是说前面所说的TCP应答号将会包含在TCP数据包中；有两个取值：0和1，为1的时候表示应答域有效，反之为0；
    • PSH：这个标志位表示Push操作。所谓Push操作就是指在数据包到达接收端以后，立即传送给应用程序，而不是在缓冲区中排队；
    • RST：这个标志表示连接复位请求。用来复位那些产生错误的连接，也被用来拒绝错误和非法的数据包；
    • SYN：表示同步序号，用来建立连接。SYN标志位和ACK标志位搭配使用，当连接请求的时候，SYN=1，ACK=0；连接被响应的时候，SYN=1，ACK=1；这个标志的数据包经常被用来进行端口扫描。扫描者发送一个只有SYN的数据包，如果对方主机响应了一个数据包回来 ，就表明这台主机存在这个端口；但是由于这种扫描方式只是进行TCP三次握手的第一次握手，因此这种扫描的成功表示被扫描的机器不很安全，一台安全的主机将会强制要求一个连接严格的进行TCP的三次握手；
    • FIN： 表示发送端已经达到数据末尾，也就是说双方的数据传送完成，没有数据可以传送了，发送FIN标志位的TCP数据包后，连接将被断开。这个标志的数据包也经常被用于进行端口扫描。

建立连接和断开连接

建立连接的过程有个更常用的名字叫做“三次握手”，就是说建立连接的过程发送了三个数据报；断开连接也叫做“四次握手”，是说这个过程发送了四个数据报。下面我们通过一张图来看看具体细节（其中一条线代表一次握手）:

上图还是比较容易看懂的，但是可能你会有个疑问，为什么建立连接只需三次握手，关闭连接却要四次握手？或者说在建立连接的时候服务端可以将ACK+SYN一起返回，而断开连接的时候却不能将ACK+FIN一起返回？因为接收端在收到发送端发送过来的FIN时并不能立即关闭SOCKET（SOCKET就是在应用层和传输层之间的一个抽象层，它把TCP/IP层复杂的操作抽象为几个简单的接口供应用层调用，已实现进程在网络中通信），因为它的数据还没有完全传给了上面的应用层。所以只能先回复一个ACK报文，告诉Client端，”你发的FIN报文我收到了”。只有等到我Server端所有的报文都发送完了，我才能发送FIN报文，因此不能一起发送。故需要四步握手。

以段为单位

TCP在传送大量数据时，是以最大消息长度MSS的大小将数据进行分割传送。进行重发也是以MSS为单位的。MSS是在三次握手的时候，在两端主机之间被计算得出。两端的主机在发出建立连接的请求时，会在TCP首部中写入MSS选项，告诉对方自己的接口能够适应的MSS的大小。然后在两者之间选择一个较小的值投入使用（如果某一方的MSS选项被忽略，则数据报长度不超过576字节，其中IP首部20字节，TCP首部20字节，MSS 536字节）。

窗口控制

前面的以段为单位传输有一个缺点：每发一个段就停下来进行一次确认应答的处理。这样网络的吞吐量就会下降。最容易想到的就是采用流水线，所以这里也一样，采用滑动窗口的方式并行处理。窗口大小就是指无需等待确认应答而可以继续发送的最大值。这个机制的实现需要使用大量的缓冲区，实现对多个段同时进行确认应答。在发送端收到确认应答之前，数据都需暂时保存到缓冲区中。当收到确认应答后，将缓冲区的数据清除，然后窗口向前滑动到还未进行确认应答的位置。

    流控制

在TCP的首部中，专门有个字段是用来通知当前空闲窗口大小的，发送端根据这个字段的指示，对发送数据的量进行控制，这就是TCP的流量控制。

拥塞控制

来到了最难的一部分。一般来说，计算机网络处于一个共享的环境，在网络拥堵的情况下，突然发送一个很大的数据报，极有可能导致整个网络的瘫痪。所以TCP中有拥塞控制这一机制。

拥塞控制主要是四个算法：1）慢启动，2）拥塞避免，3）拥塞发生，4）快速恢复。这四个算法不是一天都搞出来的，这个四算法的发展经历了很多时间，到今天都还在优化中。

慢启动 - Slow Start

慢启动的算法如下(cwnd全称Congestion Window)：

  1. 连接建好的开始先初始化cwnd = 1，表明可以传一个MSS大小的数据。
  2. 每当收到一个ACK，cwnd++; 呈线性上升
  3. 每当过了一个RTT，cwnd = cwnd*2; 呈指数让升

  4. 还有一个ssthresh（slow start threshold），是一个上限，当cwnd >= ssthresh时，就会进入“拥塞避免算法”（后面会说这个算法）

    拥塞避免算法 – Congestion Avoidance

    前面说过，还有一个ssthresh（slow start threshold），是一个上限，当cwnd >= ssthresh时，就会进入“拥塞避免算法”。一般来说ssthresh的值是65535，单位是字节，当cwnd达到这个值时后，算法如下：

  5. 收到一个ACK时，cwnd = cwnd + 1/cwnd

  6. 当每过一个RTT时，cwnd = cwnd + 1

这样就可以避免增长过快导致网络拥塞，慢慢的增加调整到网络的最佳值。很明显，是一个线性上升的算法。

拥塞状态时的算法

前面我们说过，当丢包的时候，会有两种情况：

  1. 等到RTO超时，重传数据包。TCP认为这种情况太糟糕，反应也很强烈。

sshthresh = cwnd /2
cwnd 重置为 1
进入慢启动过程

  1. Fast Retransmit算法，也就是在收到3个duplicate ACK时就开启重传，而不用等到RTO超时。

TCP Tahoe的实现和RTO超时一样。
TCP Reno的实现是：
cwnd = cwnd /2
sshthresh = cwnd
进入快速恢复算法——Fast Recovery
上面我们可以看到RTO超时后，sshthresh会变成cwnd的一半，这意味着，如果cwnd<=sshthresh时出现的丢包，那么TCP的sshthresh就会减了一半，然后等cwnd又很快地以指数级增涨爬到这个地方时，就会成慢慢的线性增涨。我们可以看到，TCP是怎么通过这种强烈的震荡快速而小心得找到网站流量的平衡点的。

快速恢复算法 – Fast Recovery

快速重传和快速恢复算法一般同时使用。快速恢复算法是认为，你还有3个Duplicated Acks说明网络也不那么糟糕，所以没有必要像RTO超时那么强烈。 注意，正如前面所说，进入Fast Recovery之前，cwnd 和 sshthresh已被更新：

  • cwnd = cwnd /2
  • sshthresh = cwnd
    然后，真正的Fast Recovery算法如下：
  • cwnd = sshthresh + 3 * MSS （3的意思是确认有3个数据包被收到了）
  • 重传Duplicated ACKs指定的数据包
  • 如果再收到 duplicated Acks，那么cwnd = cwnd +1
  • 如果收到了新的Ack，那么，cwnd = sshthresh ，然后就进入了拥塞避免的算法了。
